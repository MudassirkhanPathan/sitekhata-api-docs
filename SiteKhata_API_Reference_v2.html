1 Million Users — Monolithic Scale Architecture
Node.js Monolith · PostgreSQL · Redis · PM2 Cluster · Nginx · AWS
★ Core Design
⚡ Why Monolith first? (Not Microservices)
SiteKhata ke use case ke liye monolith sahi choice hai. 1M users = ~5000 concurrent users at peak. Ek well-optimized Node.js monolith + Redis cache + read replicas is easily 5-10M users tak handle kar sakta hai bina microservices complexity ke. Microservices tabhi karo jab team 20+ engineers ho aur deployment complexity justify ho.
🌐
LAYER 1 — EDGE & CDN
Cloudflare CDN — static assets, DDoS protection, SSL termination
Rate Limiting at edge — 100 req/min per IP
Geo-routing — India traffic → Mumbai region
Bot protection — challenge suspicious OTP requests
⚖️
LAYER 2 — LOAD BALANCER
Nginx (or AWS ALB) — round-robin to app instances
Health checks — /health endpoint, remove unhealthy pods
Sticky sessions — NOT needed (stateless JWT)
WebSocket upgrade — for activity timer live updates
🖥️
LAYER 3 — APP SERVERS
Node.js + Express monolith, PM2 cluster mode — 1 process per CPU core
Horizontal scaling — 4-8 EC2 instances (t3.medium to c5.xlarge)
Auto-scaling group — CPU > 70% → add instance
Zero-downtime deploy — rolling updates via PM2
🗄️
LAYER 4 — DATABASE
PostgreSQL Primary — all writes
2x Read Replicas — GET APIs, reports, dashboard
PgBouncer — connection pooling, max 100 connections per instance
Indexes — siteId, userId, date, type on transactions table
⚡
LAYER 5 — CACHE (REDIS)
Redis Cluster (ElastiCache) — 3 shards for HA
OTP storage — otp:{phone} TTL 300s
Dashboard cache — dash:{userId} TTL 60s
Site stats cache — stats:{siteId} TTL 30s, invalidate on new tx
📨
LAYER 6 — ASYNC JOBS
Bull Queue (Redis-backed) — PDF generation, SMS sending
Worker process — separate Node.js process for heavy jobs
Cron jobs — reminder push notifications (daily 8AM)
Dead letter queue — failed jobs retry 3x then alert
☁️
LAYER 7 — STORAGE
AWS S3 — PDF reports storage
Presigned URLs — 1 hour expiry, direct client download
S3 Lifecycle rules — delete reports older than 90 days
CloudFront — in front of S3 for fast PDF downloads

🚀1M Scale NotesToken security at scale▼
JWT STRATEGY
Short-lived accessToken (15 min) — stateless, no DB lookup needed on every request
refreshToken rotation — new token every refresh, old token invalidated. Prevents token theft.
Store only bcrypt hash of refreshToken in DB — not plaintext
DB UPSERT PERFORMANCE
ON CONFLICT DO UPDATE is single atomic query — no race condition possible
Index on phone column (UNIQUE) — O(1) lookup even at 1M users
Update logged_in_at — useful for analytics and inactive user cleanup

🚀1M Scale NotesAsync heavy job pattern▼
WHY BULL QUEUE?
Puppeteer PDF generation takes 3-10 seconds. Synchronous response would timeout mobile clients.
Bull Queue with Redis backend = job persists even if app server crashes mid-generation
Scale worker pods independently from API pods — PDF workers are CPU-heavy
PUPPETEER AT SCALE
Run Puppeteer in separate Docker container (dedicated worker), not in main API process
Browser pool: Keep 2-3 Puppeteer instances warm (browser launch is slow ~2s)
S3 cache: same report requested within 1hr → return cached URL, skip regeneration
